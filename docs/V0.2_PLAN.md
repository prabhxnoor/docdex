# docdex v0.2 — "Trust & Context Foundations"

Status: **planning → in progress** (2026-06-11)
Inputs: independent Codex audit (`AUDIT_REPORT.md`) + context-efficiency
architecture review (`CONTEXT_EFFICIENCY_ARCHITECTURE_REPORT.md`), both run
against commit `49f9330` (v0.1.1). The four highest-severity findings were
re-reproduced by hand before planning.

## The objective, restated

docdex exists to **give an AI agent the majority of the context it needs to do
a task inside a document-heavy project, with the fewest tokens** — invoke `ctx`,
get the relevant context with high accuracy every time, and don't make the model
re-read 5,000 files or re-derive context into its own memory each session.

That reframes the target. v0.1 measured "can it find one hidden fact?" The real
metric is **task-context recall at a token budget**: did the agent get *enough*
context to finish the job, with citations, while wasting few tokens? The product
is ultimately a **context packet**, not a search hit list.

## Decisions locked (with the user)

1. **Security fixes fold into v0.2** (no separate patch release), but the
   CHANGELOG is maintained as a first-class, human-readable record.
2. **Scope = hardening + context-router foundations.** Ship the trust fixes,
   a real ranking engine, and a first `ctx context` packet command — defer the
   neural layers to v0.3.
3. **SQLite FTS5 becomes the query engine** (it's in the Python standard
   library — no new dependency). The extracted `.txt` caches remain the
   inspectable source of truth; the database is a rebuildable index.
4. **docdex never calls an LLM itself.** It stays deterministic and hands the
   already-running agent (Claude/Codex) a structured packet to reason over.
   No API keys, no network, reproducible.

## Workstreams

### A. Trust & correctness (do first — "wrong context is worse than none")

| ID | Fix | Audit |
|---|---|---|
| A1 | Confine `index_dir` to a safe in-project folder name; guard every purge delete | DDX-001 (CRITICAL, data loss) |
| A2 | Skip symlinked files by default; opt-in `follow_symlinks`, target must stay in-project | DDX-002 (CRITICAL, privacy) |
| A3 | `semantic`: validate query (exit 2), threshold weak hits to "no match" (exit 1), index short files | DDX-003, DDX-009 (CRITICAL) |
| A4 | Validate external embedder output (finite numbers, stable dimension); clean errors | DDX-004 (MAJOR) |
| A5 | Corrupt `.docdex.json` / inventory → friendly message + clean exit, no traceback | DDX-008 (MAJOR) |
| A6 | A duplicate file counts as new, not renamed | DDX-006 (MAJOR) |

### B. Retrieval engine upgrade

- **B1.** SQLite state DB (`_state/index.db`) beside the caches: `files`,
  `chunks` (with token counts, offsets, stable IDs). DB is rebuildable from the
  `.txt` caches; caches stay the audit-able source of truth.
- **B2.** FTS5 + BM25 powering `search` (same CLI/UX). Fixes term-stuffing
  (DDX-007) and the per-query linear-scan cost that degrades past ~10k files.
- **B3.** Real token accounting per chunk (`tiktoken` if installed, chars/4
  fallback) — needed for honest budgets.
- **B4.** Honest fuzzy branding: keep the dependency-free hash backend as the
  local fallback, make real `semantic` shine when `DOCDEX_EMBED_CMD` is set,
  and fuse lexical + vector with Reciprocal Rank Fusion when both exist.

### C. The product step — `ctx context`

- **C1.** `docdex context "task" --budget N` → a deterministic, token-budgeted
  **evidence packet**: task restatement, index freshness, ranked answer-bearing
  excerpts with source path + location, running token estimate, and an explicit
  **"Missing / weak evidence"** section. Heuristic packing: dedupe, prefer
  source diversity, stop when the budget or confidence threshold is hit.
- **C2.** `--explain` shows the sub-queries, candidates, scores, and what was
  dropped — for trust and debugging.
- **C3.** Update the scaffolded `CLAUDE.md`/`AGENTS.md` so agents do
  `status → context --budget → fill only the gaps` instead of bulk-loading.

### D. Prove it — task benchmark

- **D1.** New **Suite B (form-filling)**: a synthetic vendor/client corpus
  (contracts, sheets, decks, scanned PDFs + notes) and a multi-field form.
  Measure **field accuracy, evidence recall, missing-field honesty, total
  tokens, and number of `ctx` calls** for `ctx context` vs read-everything vs a
  naive search loop.
- **D2.** Keep Suite A (single hidden fact) as a regression guard; add
  term-stuffing and no-answer cases.
- **D3.** Report real-tokenizer numbers (the audit showed 36× becomes ~30× under
  a real tokenizer — we cite the real number).

### E. Minor fixes, docs, release

DDX-005 (document `status` as a cheap heuristic; finer mtime where practical),
DDX-010 (wrapper-from-subdirectory docs), DDX-011 (don't count scaffold READMEs
as vision work), DDX-012 (`dumps` before sync → clear message), DDX-013
(huge-text-file cap or explicit doc), DDX-014 (SPDX license metadata). Tag
v0.2.0, push, verify a clean install.

## Explicitly deferred to v0.3+

Contextualized chunks (document summaries/entities prepended before indexing),
a neural reranker, a structured fact-extraction layer with `ctx facts` /
`ctx fill-context` and a field registry, source-authority configuration, an
ANN/vector store for 100k+ files, and any Spotlight integration. These need the
foundation above and a real task benchmark to evaluate against first.

## Sequencing

A → B → C → D → E. A is independent and ships trust immediately. B underpins C.
D validates C against the real objective. E closes out and releases. The
CHANGELOG's `[Unreleased]` section is updated as each item lands.
